## 2. Release Workflow (`release.yml`)

**Trigger:** push of a `v*` tag, or manual `workflow_dispatch`  
**Status:** ŌØī All 66 runs failed (every release since workflow was created)

### Failure root cause ŌĆö macOS `EEXIST` hardlink error

```
Ō©» EEXIST: file already exists,
  link '.../build/icon.icns'
    -> '.../release/mac/Tesserin.app/Contents/Resources/icon.icns'
```

**Why it happens:**  
`package.json` defines two Mac targets for two architectures:

```json
"mac": {
  "target": [
    { "target": "dmg", "arch": ["x64", "arm64"] },
    { "target": "zip", "arch": ["x64", "arm64"] }
  ]
}
```

When `electron-builder --mac` runs, it processes `dmg` and `zip` for the same architecture **concurrently**. Both use the same intermediate staging directory (`release/mac` for x64, `release/mac-arm64` for arm64). Both attempt to `link(build/icon.icns, release/mac/Tesserin.app/Contents/Resources/icon.icns)` at the same time; the second call fails with `EEXIST` because the hardlink was already created by the first.

**Fix applied in this PR:**  
Both `release.yml` and `ci.yml` now pass `--arm64` explicitly for macOS so only one architecture is packaged per run, eliminating the concurrent-write race:

```yaml
if [ "${{ matrix.platform }}" = "mac" ]; then
  pnpm electron-builder --mac --arm64 --publish never
else
  pnpm electron-builder --${{ matrix.platform }} --publish never
fi
```

## 3. What Gets Ignored in the Build

### `.gitignore` ŌĆö build artefacts excluded from git

| Pattern | What it covers |
|---------|---------------|
| `node_modules/` | pnpm dependency tree |
| `dist/` | Vite renderer output (`pnpm build`) |
| `electron/dist/` | TypeScript-compiled Electron main-process files (`pnpm tsc -p electron/tsconfig.json`) |
| `release/` | Electron-builder packaged apps and installers |
| `*.dmg` | macOS disk image |
| `*.AppImage` | Linux AppImage |
| `*.snap` | Linux Snap package |
| `*.deb` | Debian package |
| `*.rpm` | RPM package |
| `*.exe` | Windows NSIS installer |
| `*.msi` | Windows MSI installer |
| `*.blockmap` | Differential update blockmap |
| `*.yml` _(except `.github/workflows/*.yml`)_ | Auto-updater manifests (`latest-mac.yml`, `latest.yml`, `latest-linux.yml`). These are generated by electron-builder and live only in GitHub Releases, not in the repository. |
| `.env`, `.env.*` _(except `.env.example`)_ | Secrets and local environment config |
| `.DS_Store`, `Thumbs.db`, `Desktop.ini` | OS metadata |

### Electron-builder ŌĆö `files` field in `package.json`

Only these paths are bundled into the packaged Electron app:
```
dist/**/*          ŌåÆ Vite renderer output
electron/dist/**/* ŌåÆ compiled Electron main process
```
Everything else (source TypeScript, tests, docs, `tesserin-mcp-server/`, etc.) is excluded from the installed app.

## 4. MCP Server ŌĆö Windows (and Other Platforms)

### Docker MCP (primary path)

`mcp.json` registers Tesserin in the Docker MCP catalog. The `command` / `args` reference paths **inside the Docker container** built from `Dockerfile.mcp`.

**Bug fixed in this PR (`Dockerfile.mcp`):**  
The TypeScript compiler outputs Electron main-process files to `electron/dist/` (per `electron/tsconfig.json ŌåÆ outDir: ./dist`), but the previous `Dockerfile.mcp` was trying to copy from `dist-electron/` (which never existed), so the Docker MCP container would start and immediately fail with `Cannot find module 'dist-electron/main.js'`.

```dockerfile
# Before (wrong ŌĆö path does not exist after tsc)
COPY --from=builder /app/dist-electron ./dist-electron

# After (correct ŌĆö matches electron/tsconfig.json outDir)
COPY --from=builder /app/electron/dist ./dist-electron
```

The `ENTRYPOINT ["node", "dist-electron/main.js", "--mcp"]` and the `mcp.json` args stay unchanged because they reference the path **inside the container**, which is now correctly populated.

### Native MCP (Windows / macOS / Linux ŌĆö no Docker)

For users who want to run the MCP server natively (outside Docker), the compiled binary is at:

```
electron/dist/main.js   ŌåÉ correct path after: pnpm tsc -p electron/tsconfig.json
```

The `npm run mcp` script in `package.json` already uses the correct path:
```json
"mcp": "tsc -p electron/tsconfig.json && node electron/dist/main.js --mcp"
```

On **Windows**, Claude Desktop / other MCP clients must be configured to call:
```json
{
  "command": "node",
  "args": ["C:\\path\\to\\Tesserin-pro\\electron\\dist\\main.js", "--mcp"]
}
```

The `mcp.json` Docker-catalog schema is not used for native connections; users configure their MCP client directly.

### Why MCP is "ignored" in the Windows build

Electron-builder's `files` field only includes `dist/**/*` and `electron/dist/**/*`. The MCP server entry-point (`electron/dist/main.js`) **is** included when the app is packaged, and it supports `--mcp` mode at runtime. There is no Windows-specific exclusion ŌĆö the `--mcp` flag works identically on Windows, macOS, and Linux once the app is built.

What users reported as "MCP being ignored on Windows" is most likely:
1. The wrong path in `mcp.json` / `Dockerfile.mcp` (fixed above).
2. No `latest-win.yml` auto-updater manifest in recent releases (because the `create-release` job was always skipped due to the macOS build failure). Windows users running v1.0.1 would not receive auto-update prompts for v1.2.0.
